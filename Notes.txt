## Three.js Optimization Context (Project Summary)

I’m building a Three.js web app that:

* Renders maps using OSM data
* Displays animated clouds (instanced)
* Procedurally generates large numbers of street lamps (hundreds to thousands)

---

## Current Performance Concerns

### 1. Cloud animation (per-frame updates)

* Using `InstancedMesh`
* Updating matrices every frame on CPU
* Previously used `Object3D` per instance (now optimizing away)

**Key optimizations:**

* Reuse math objects (`Vector3`, `Matrix4`, etc.) instead of allocating per frame
* Replace `Object3D.updateMatrix()` with `Matrix4.compose()`
* Avoid unnecessary material updates
* Potential future: move animation to GPU (vertex shader)

---

### 2. Shadows (DirectionalLight / sun)

* Currently using a single shadow map
* Considering Cascaded Shadow Maps (CSM)

**Conclusion:**

* CSM improves quality, not performance
* Adds multiple shadow passes (worse for FPS)
* Better approach:

  * Reduce shadow map size
  * Tighten shadow camera bounds
  * Disable shadows for clouds
  * Update shadows less frequently

---

### 3. Street lamp LOD (distance-based visibility)

Current approach:

* Per-lamp distance check every ~15 frames
* Toggle `mesh.visible`

**Problems:**

* Per-object visibility toggling is expensive at scale
* Doesn’t scale well to thousands of lamps

---

## Recommended Optimization Strategy

### Best solution (major improvement)

Switch lamps to `InstancedMesh`:

* Single draw call
* Control visibility via `mesh.count` instead of toggling objects
* Rebuild visible instance list based on distance

---

### If keeping individual meshes

* Ignore Y axis in distance checks
* Cache lamp positions (avoid property access)
* Add hysteresis (reduce flickering + updates)
* Only update when camera moves significantly

---

### For large-scale scenes

* Use spatial partitioning (grid / quadtree)
* Only evaluate nearby lamps instead of all

---

## Key Takeaways

* **Instancing is critical** for both clouds and lamps
* **Avoid per-frame allocations** and scene graph changes
* **CSM = better visuals, worse performance**
* **Distance checks scale poorly without spatial partitioning**
* Biggest wins come from:

  1. InstancedMesh
  2. Reducing draw calls
  3. Minimizing per-object CPU work
